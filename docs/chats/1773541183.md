# Session Summary

Date: 2026-03-14

## Context

This session focused on closing parity gaps in the V2 rasterizer refactor,
especially around intercept-stage behavior, testability of staged rendering,
artifact modeling, and the standard for accepting parity and numeric tests.

## Decisions

- Legacy rasterizer behavior is the baseline. Any V2 behavior difference should
  be treated as a parity gap unless the legacy contract is explicitly being
  changed.
- Do not hide uncertainty behind loose tolerances. Numeric assertions must use
  tight, justified bounds tied to the contract being checked rather than broad
  "make it pass" epsilons.
- Build V2 rasterizers to a shared staged interface so intercepts and waveform
  generation are first-class phases:
  `renderIntercepts(Artifacts&)`, `renderWaveform(Artifacts&)`, and
  `renderArtifacts(Artifacts&)`.
- Use one shared artifact struct across rasterizers. Keep specialized outputs
  such as `colorPoints` as optional fields on that shared struct rather than
  introducing artifact inheritance.
- Do not add redundant artifact count fields when the containers already expose
  `size()`. Container or buffer size is the source of truth.
- Prefer real pipeline-stage testing over test-only seams or duplicated oracle
  logic when validating intercept generation and point-positioner behavior.
- Keep `cycle` tests in the peer `cycle/tests` directory rather than under
  `cycle/src/.../tests`.
- For this MR, do not use an environment-variable gate for the V2 path. The
  acceptance signal is that the full application builds and launches cleanly.

## Generalized Rules

- When refactoring a legacy DSP pipeline, preserve the observable stages in the
  new architecture if those stages are independently consumed, tested, or
  updated by other systems.
- If a visual or downstream artifact is only produced by one rasterizer family,
  model it as an optional channel on shared artifacts before introducing a new
  type hierarchy.
- Tests should be necessary and sufficient: strict enough to fail on real
  behavioral drift, but written against the actual contract rather than against
  incidental implementation details.
- For envelope behavior, audio-level tests should cover looping and release
  behavior directly; these are the tests most likely to catch incorrect
  positioner or intercept-stage assumptions.
- In aligned or hot-path rasterizer code, continue using the project’s
  `ScopedAlloc` and `Buffer` constructs rather than tidier standard containers
  that do not preserve the required memory behavior.

## Open Questions

- Remaining work to remove active legacy fallback paths and complete final
  production call-site migration to the staged V2 rasterizer interface.
