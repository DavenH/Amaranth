# Session Summary

Date: 2026-03-14

## Context

This session focused on cross-platform unit-test failures in `Buffer`,
`VecOps`, and `ConvReverb`, especially differences between the Linux IPP path
and the macOS Accelerate path.

## Decisions

- When Linux and macOS disagree, prefer fixing the implementation at the source
  over forcing tests to pass with platform-specific expectations.
- If a failing test appears to reflect a real test bug rather than an
  implementation bug, surface that explicitly for user review before changing
  the test to pass.
- In DSP, math, and platform-accelerated code paths, prefer Accelerate/IPP
  library primitives over handwritten scalar or loop-based replacements.
- If a custom math replacement is temporarily needed to prove behavior, follow
  up by finding the library-equivalent formulation and use that instead.
- Platform-specific representation details such as packed FFT layouts should be
  normalized behind API boundaries like `Transform::getComplex()` rather than
  leaking into higher-level algorithms.

## Generalized Rules

- Treat cross-platform test mismatches as implementation-contract problems
  first, not as an excuse to relax tests.
- Keep non-commutative accelerated operations explicit per function and per
  type instead of relying on one broad operand-order macro.
- Encapsulate backend-specific data layout quirks at subsystem boundaries so
  callers can operate on canonical representations.
- When evaluating a possible test fix, distinguish ephemeral harness mistakes
  from durable behavioral contracts and ask the user to review the former.
- Prefer library-backed vectorized formulations for windowing and related DSP
  primitives, even when a handwritten equivalent is easy to derive.

## Open Questions

- Whether the remaining `ConvReverb` failure is an implementation bug in the
  Accelerate FFT/convolution path or a stale-data bug in the test harness.
- Whether `VecOps` should gain explicit `Complex64` coverage in addition to the
  current `Complex32` tests.
