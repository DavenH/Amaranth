# Session Summary

Date: 2026-03-14

## Context

This session focused on debugging `oscillo` real-time pitch tracking on macOS,
especially the `RealTimePitchTracker` cycle-diff path, the pitch-debug heatmap,
and the mismatch between apparent score minima and the note ultimately selected
in the UI.

## Decisions

- When a DSP heuristic appears wrong in live use, prefer instrumenting the real
  score landscape first rather than continuing to tune blindly.
- Visual diagnostics such as the pitch heatmap are considered authoritative
  enough to challenge the current selection logic when they disagree.
- Temporary debug output is acceptable if it helps inspect live scoring, such
  as CSV dumps of `periodScores` at a throttled cadence.
- Keep tracker test behavior intact while refining live heuristics; if a change
  improves the live display but breaks established tracker tests, treat that as
  a signal to re-evaluate the approach rather than forcing it through.
- If UI-side latching or voting can explain an apparently wrong DSP result,
  investigate and patch that path instead of assuming the scorer is solely at
  fault.

## Generalized Rules

- For realtime DSP debugging, add observability before adding more heuristics:
  prefer score dumps, overlays, or plots over speculative retuning.
- Treat disagreements between debug visualizations and final UI behavior as a
  pipeline problem first: verify scorer, gating, octave resolution, and UI
  latching separately.
- In hot DSP paths, keep normalization or compensation math precomputed outside
  the scoring loop whenever possible.
- Temporary debug instrumentation may remain long enough to validate live
  behavior, but it should stay lightweight and easy to remove.
- Preserve regression tests while tuning heuristics; use failing tests as a
  guardrail against overfitting to one live input case.

## Open Questions

- Whether the remaining note-selection error is primarily in the cycle-diff
  scoring heuristic, the octave-resolution rule, or the onset/vote/latch path
  above the tracker.
- Whether the occasional dark "hole" in the pitch-debug heatmap is caused by
  transient low-confidence frames, buffer timing, or display normalization.
