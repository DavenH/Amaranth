# Date

2026-07-16

# Context

Realtime voice rasterization needs larger storage when editable mesh topology
grows, but the application has no fixed authoring topology limit.

# Decisions

- Do not impose a musical mesh-topology ceiling merely to simplify allocation.
- Prepare enlarged voice rasterizer state off the audio thread.
- Include an immutable mesh snapshot in the prepared replacement; buffer-only
  replacement is insufficient while editor meshes remain mutable.
- Publish prepared replacements immutably and let active voices retain their
  last valid state until a safe adoption boundary.

# Generalized Rules

- Prefer immutable off-thread replacement over resizing active realtime state.
- Preserve current audio behavior while replacement state is being prepared.
- Use defensive corruption limits separately from ordinary authoring limits.
