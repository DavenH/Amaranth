# Session Summary

Date: 2026-03-14

## Context

This session focused on investigating NaNs in legacy and V2 rasterizer tests
during the V2 rasterization refactor. Some failing tests turned out to be
mixing product contracts, such as driving legacy env logic through the graphic
slot or feeding legacy voice rasterization meshes with out-of-contract phase
data.

## Decisions

- Non-finite output is never acceptable in either the legacy or V2 rasterizer
  pipelines. Zero NaNs and infinities is a hard requirement, not a soft parity
  tolerance.
- Before treating a parity failure as a rasterizer bug, align the test harness
  with the real production usage contract for that rasterizer.
- Legacy env runtime behavior should be tested through the stateful runtime
  slot, not the graphic preview slot.
- Legacy voice parity tests should use meshes that satisfy the mesh contract
  expected by the legacy code, including non-negative normalized phase values.
- Once finiteness is restored, treat any remaining parity failure as a real
  behavioral gap rather than sanitizing it away.

## Generalized Rules

- Treat finiteness as a first-class contract: assert that legacy and V2
  rasterizers produce finite curves, wave buffers, and rendered samples before
  comparing behavior.
- Align every test with the actual subsystem policy it is supposed to model.
  Do not reuse a convenient but semantically different path just because it is
  easier to exercise.
- When a parity test fails, separate three questions in order:
  1. Is the test driving the code the way production does?
  2. Is the oracle itself finite and domain-valid?
  3. Only then, does V2 still diverge from legacy behavior?
- Do not keep sanitization in place once the underlying finiteness bug is
  fixed; tighten the test back to hard finite-output assertions.

## Open Questions

- Whether the remaining large V2-vs-legacy voice parity delta is due to
  intercept/chaining policy differences, curve padding differences, or sampling
  state progression after the NaN fixes.
