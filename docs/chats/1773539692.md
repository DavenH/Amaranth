# Session Summary

Date: 2026-03-14

## Context

This session focused on tightening the V2 rasterizer architecture during the
realtime pipeline refactor, especially ownership of block rendering, elimination
of duplicated orchestration, and fidelity to legacy voice chaining/sampling
semantics.

## Decisions

- Do not duplicate render-path boilerplate across concrete rasterizers.
  Shared block-render orchestration belongs in the common V2 pipeline layer.
- Prefer explicit pipeline phases such as `renderIntercepts`,
  `renderWaveform`, and `renderBlock` over ad hoc per-rasterizer `renderAudio`
  implementations that repeat the same setup/sampling flow.
- Do not keep APIs that only expose intermediate waveform arrays unless there is
  a real caller that needs them. Intermediate artifact access is acceptable for
  staged internals and tests, but the primary runtime path should be the shared
  block-render path.
- Avoid large repeated argument lists for stage orchestration. If related values
  already live in a cohesive object such as the rasterizer workspace or artifact
  bundle, centralize that wiring instead of re-threading the same parameter set
  through multiple callers.
- Legacy voice chaining/sample behavior is the baseline: do not wrap local phase
  inside the per-sample V2 voice render loop. Match legacy spillover behavior
  instead of inventing cycle-boundary wrapping in the chaining sampler.

## Generalized Rules

- Put orchestration once in the highest common layer that owns the behavior.
  Concrete rasterizers should only provide the parts that are genuinely unique,
  such as stateful sampling semantics.
- Treat repeated code as a design failure unless there is a documented semantic 
- difference that cannot be expressed through a shared hook.
- When validating parity for voice/chaining behavior, compare against the real
  legacy sampling model, not an intuitively "cleaner" waveform-looping model.
- Do not introduce local phase wrapping in chaining samplers unless legacy code
  demonstrably did so.
