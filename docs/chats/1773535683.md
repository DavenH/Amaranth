# Session Summary

Date: 2026-03-14

## Context

This session focused on startup debugging, singleton initialization order,
updater graph validation, naming cleanup around guide curves, and the user's
preferred way to preserve decision patterns for future agents.

## Decisions

- The user wants startup/debugging work carried through until launch is clean,
  unless a genuinely ambiguous choice appears.
- `order = 0` is acceptable for singletons with no startup dependencies beyond
  the `< 0` bucket.
- When a getter name is misleading, rename it to reflect the real abstraction
  rather than preserving legacy terminology.
- The migration direction for the old `deformer` concept is:
  `GuideCurve` for the authored layer object,
  `GuideCurveProvider` for the sampled/evaluation service,
  `GuideCurvePanel` for the UI/editor.
- Consistency matters enough that broad renames, including filenames, are
  preferred over partial compatibility naming.
- Temporary debug accessors are acceptable, but they should be clearly marked
  as temporary and removable after debugging.
- Before resuming larger architectural cleanup, the user prefers stabilizing
  the current runtime path first.

## Generalized Rules

- If the app is crashing during startup, prefer end-to-end stabilization over
  stopping at diagnosis.
- When the user has a naming migration in progress, align terminology across
  type names, accessors, and filenames rather than leaving mixed legacy names.
- Distinguish provider/service objects from selected content or layer data in
  naming and architecture.
- Treat validator/debug instrumentation as temporary unless the user asks to
  productize it.
- If a system is incomplete but directionally clear, the user prefers an
  incremental cleanup that first makes the app launch reliably.
- If a singleton or updateable is conceptually part of startup invariants, the
  user expects explicit validation rather than silent omission.

## User Preference Signals

- The user values deterministic startup behavior and explicit dependency
  understanding.
- The user is comfortable with narrow defensive guards during debugging, but
  likely wants a later pass to separate durable lifecycle fixes from temporary
  null-protection.
- The user prefers agents to accumulate and reuse these patterns so future work
  can be more autonomous.

## Open Questions

- Which temporary startup guards should remain as permanent hardening, and which
  should be removed once attachment order is formalized?
- Whether this archive should later be complemented by a top-level `AGENTS.md`
  summary section for especially durable cross-session rules.
