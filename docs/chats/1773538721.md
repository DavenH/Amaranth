# Session Summary

Date: 2026-03-14

## Context

This session focused on making the resampling path build and run on macOS/Apple
Silicon, removing hard Intel IPP assumptions, and preserving equivalent
behavior across IPP and Accelerate/JUCE backends.

## Decisions

- A low-quality linear resampler is not an acceptable primary non-IPP backend;
  if a fallback is needed, it should sit below a higher-quality sinc-based
  implementation.
- Prefer delegating to a proven library resampler such as JUCE's windowed-sinc
  interpolator before introducing a custom scalar resampling implementation.
- Do not let `#ifdef USE_IPP` change higher-level semantics just to work around
  a deleted operation or backend limitation.
- If direct assignment/copy of a backend object is unavailable, replace it with
  an explicit semantically equivalent operation rather than silently dropping
  behavior on one platform.
- Replace IPP math constants used outside the IPP backend with portable JUCE or
  standard-library constants.

## Generalized Rules

- Keep backend quality tiers explicit: use the best available library-backed DSP
- path first, and demote simpler interpolation to true emergency fallback only.
- Preserve cross-platform semantics at the API and call-site level even when
  backend internals differ.
- When a type becomes non-copyable because of a backend dependency, add an
  explicit state-transfer API if the surrounding code still needs equivalent
  behavior.
- Avoid scattering IPP-specific constants or assumptions into code that should
  compile under Accelerate or other backends.

## Open Questions

- Whether `Resampler::copyStateFrom()` should eventually preserve more backend
  internal state, or whether higher-level code should be refactored to avoid
  copying live resampler state entirely.
