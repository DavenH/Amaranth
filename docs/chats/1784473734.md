# Date

2026-07-19

# Context

Cycle v2 effect editors were refined through detailed feedback on Delay,
Reverb, and EQ controls and visualizations. The durable requirement is that an
effect display explain the real DSP and that its interaction model expose,
rather than conceal, parameter semantics.

# Decisions

- Keep serialized parameter IDs stable while improving user-facing names.
- Use conventional audible terminology. Explain bespoke behavior with a short
  tooltip; keep value readouts terse.
- Treat continuous and discrete controls differently. Continuous controls stay
  continuous except for proximity snapping near useful landmarks. Discrete
  controls show every meaningful stop and snap strictly.
- Place ticks and calculate snap proximity in slider-position space after the
  control's nonlinear mapping.
- Derive visualizations from the authoritative DSP implementation and preserve
  comparable visual scaling so parameter effects remain observable.
- Draw grids and axes beneath data, share exact floating-point geometry for
  related elements, and maintain symmetric margins.
- Show bypassed effect displays in greyscale while retaining the configured
  response, rather than erasing it.

# Generalized Rules

- Audit every effect parameter for its actual DSP range, mapping, quantization,
  default, unit, and causal role before designing its control.
- Make the visualization answer what each parameter audibly changes. Do not
  add decorative movement or markers that are not supported by the DSP.
- Do not normalize or post-process a visualization in ways that hide relative
  attenuation, decay, wet level, feedback, or other meaningful comparisons.
- Keep readouts compact (`6×`, `+3.2 dB`, `850 Hz`, `0.37 s`). Put explanatory
  prose in labels or tooltips.
- Give nonlinear frequency/time controls correctly mapped landmarks. Use
  pixel-distance snap zones for optional snapping, not raw-value distance.
- For EQ, prefer a logarithmic frequency axis, a strong 0 dB reference/detent,
  useful frequency landmarks, and the same filter response core as production
  DSP.
- For Reverb, ensure Size, Damping, High Pass, Wet, and Width each produce an
  honest visible consequence. Do not let visualization normalization obscure
  their effects; represent stereo Width with stereo information.
- For Delay, make the first tap reflect Wet, apply Feedback beginning with
  subsequent taps, and derive spacing and pan cycles from the production
  mappings.
- Add focused interaction fixtures for UI regressions and numerical or
  pixel-level tests for DSP-to-visualization claims. A screenshot that cannot
  demonstrate the claimed difference is not proof.
- Treat compact and expanded views as two densities of one presentation, and
  review their palette, geometry, and semantics side by side.
- Require a parameter-state sweep before accepting a visualization: endpoints,
  defaults, adjacent discrete stops, and coupled settings must be perceptually
  distinguishable where the DSP differs.
- Use shared headers and terse row ordinals for repeated controls. Do not spend
  scarce editor height repeating the same label in every row.
- Reject technically truthful visualizations that do not help a user perceive
  the parameter's effect. Remove unhelpful axes or panels instead of explaining
  them with more plot furniture.
- Treat drag latency and visual density as acceptance criteria. Keep expensive
  DSP analysis and image construction off synchronous repaint/update paths.

# Open Questions

- Choose an honest stereo representation for Reverb Width, such as paired L/R
  energy, stereo correlation, or paired spectrograms.
- Audit whether Reverb Size's power-of-two kernel quantization should be exposed
  as visible stops or replaced with a perceptually continuous presentation.
